# AGENTS.md

## 哲学

このコードベースは **将来的な設計の綺麗さを最優先** する。
短期的なビルド安定性、移行中の使いやすさ、見かけ上の後方互換性は、その目的に従属する。

「今この瞬間に動いているか」よりも、
**最終的に不純物のない構造へ到達できるか** を重視する。

- 「動けばいい」は却下。
- 「とりあえず互換レイヤーで繋ぐ」は却下。
- 「壊したくないから旧設計を残す」は却下。
- 「一旦ラップして後で消す」は却下。
- 「ビルドが赤いから削除を後回しにする」は却下。
- **「ただ最小変更でビルドを通すためだけに設計を汚す」は最も強く却下する。**
- **その場しのぎの継ぎ足しで構造を歪めるくらいなら、1から設計し直す方が正しい。**

**間違った構造を温存するくらいなら、先に壊す方が正しい。**
設計移行の途中でコードベースが壊れること、テストが落ちること、ビルドが赤くなることは許容する。
それらは失敗ではなく、**どこが旧設計に依存しているかを可視化するための情報** である。

コミットは「常に出荷可能な完成品」である必要はない。
コミットは **設計上の意図を前進させた、可逆でない構造変更の記録** であればよい。
重要なのは各コミットの緑性ではなく、**誤った抽象を一刻も早く消し、正しい抽象だけを残す流れになっていること** である。

## 最上位原則

判断に迷った場合は、以下の優先順位に従うこと。

1. **将来的な設計の純度**
2. **型・構造による不正状態の排除**
3. **旧設計の早期削除**
4. **可読性・明快さ**
5. **性能**
6. **一時的なビルド安定性**
7. **一時的な移行容易性**

短期的な都合のために 1〜4 を損なうことを禁ずる。
**「最小変更でとりあえず通す」を理由に 1〜4 を崩すことは、単なる優先順位違反ではなく、設計方針への直接的な背反である。**

## 理想構造の定義

このコードベースにおいて「理想構造」とは、単に主観的に美しく見える構造を意味しない。
以下の条件を満たす構造のみを、理想構造と呼ぶ。

- **不正状態が型または構築手順で排除されていること**
- **状態遷移が単一路または明示的な state machine / TypeState で表現されていること**
- **同一責務・同一概念・同一真実の所在（single source of truth）が一箇所にのみ存在すること**
- **モジュール境界・責務境界・依存方向が一方向であり、逆流しないこと**
- **旧設計の概念名・型・分岐・互換層・残骸が新設計へ流入していないこと**
- **外部都合のデータ表現が内部ドメインを侵食していないこと**

「なんとなく綺麗」「読みやすく見える」「将来役立つかもしれない」といった感覚的判断だけを根拠に、
理想構造を主張してはならない。
理想構造は **判定可能な条件** によってのみ定義される。

局所的に便利であっても、上記条件を破る構造は理想構造ではない。
そのような構造は一時的に動作しても、将来の整合性を破壊するため採用してはならない。

## 言語・ツール

- **実装言語は原則 Rust。** Rust で実現できる機能を他言語で書くことを禁ずる。
- スクリプト・ツール・CLI など Rust 以外が適切な場面では **TypeScript のみ** を使用する。
  **JavaScript（`.js` / `.mjs` / `.cjs`）ファイルの新規作成を禁ずる。**
  既存の JS ファイルが存在する場合は、同一 PR または後続の早い段階で TS へ移行すること。
- TypeScript コードでは `any` および型アサーション `as T` は禁止。
  信頼できない外部ソース（API レスポンス・ユーザー入力・ファイル読み込み等）は
  必ず `zod`・`valibot` 等のランタイム検証、または
  Deno 標準ライブラリの型安全パーサーを通してから型付けすること。
- JS/TS ランタイムは **Deno** を第一選択とする。
  ただし Deno での利用が困難なライブラリ（Astro 等のフレームワーク・ツールチェーンを含む）を
  使用する必要がある場合に限り、**Node.js の使用を許可する。**
  Bun の使用は禁ずる。
- Node.js を選択する場合は、その理由（「○○ が Deno 非対応のため」等）を
  `README.md` または該当の `package.json` にコメントで明記すること。
  理由なき Node.js 選択を禁ずる。
- パッケージマネージャーは **`pnpm` / `pnpm dlx` のみ** を使用する。
  `npm` / `npx` / `yarn` / `bun` コマンドの使用を一切禁ずる（Node.js・Deno を問わず）。
- `deno.json` の `compilerOptions` に以下を必ず設定し、型チェックを最大化すること。

```json
{
  "compilerOptions": {
    "strict": true,
    "noImplicitAny": true,
    "noImplicitReturns": true,
    "noFallthroughCasesInSwitch": true,
    "noUncheckedIndexedAccess": true
  }
}
```

- パッケージ管理は **Deno の import / `deno.json` の `imports` マップ** を使用。
  Node.js プロジェクトでは `pnpm` および `package.json` の `imports` マップを使用する。
  npm パッケージを Deno から使用する場合は `npm:` スキームを通じて参照すること。
- ライブラリの使用方法は **公式ドキュメントまたは一次情報** を参照すること。
  ドキュメントを確認せずに API を推測で呼び出すことを禁ずる。

## 依存関係

依存関係は実装短縮のための便利道具ではなく、構造の一部である。
したがって依存追加は、設計純度を高める場合にのみ許可する。

- **依存追加は責務分離・境界明確化・型安全性向上・内部複雑性削減に寄与する場合にのみ許可する。**
- 「早く作れる」「一時的に楽になる」「移行中だけ必要」「とりあえず動く」を理由とした依存追加を禁ずる。
- 同一責務を持つライブラリの重複導入を禁ずる。
- 一時移行のためだけのアダプタ依存・互換依存・ブリッジ依存の導入を禁ずる。
- グローバル状態を内部に隠し持つ依存、または暗黙の初期化順序に依存する依存は原則禁止とする。
- マクロによって責務境界・制御フロー・型境界を見えにくくする依存は避けること。
- **依存は責務境界の代替ではない。**
  本来内部で定義すべき構造上の整理を、外部ライブラリ導入で誤魔化してはならない。

依存を増やした時点で、その依存の思想・制約・境界がコードベースへ流入する。
したがって、依存追加はコード追加よりも重い設計判断として扱うこと。

## unsafe の禁止

- `unsafe` ブロックは原則禁止。
- やむを得ず `unsafe` が必要な場合は、事前にレビューを求め、
  安全性の根拠を詳細なコメントで明記すること。
- FFI 境界・生ポインタ操作・インラインアセンブリもすべて同扱い。
- 「unsafe でしか書けない」は設計の失敗を意味する。まず設計を見直すこと。

## エラー処理

- `unwrap()` / `expect()` の使用を **原則禁ずる。**
  「論理的に失敗しない」という主張は設計で証明するのであって、コメントで宣言するものではない。
  `unwrap()` が「安全」に見える箇所は、型設計が不十分であることのサインである。
- `panic!` をフロー制御に使うことを禁ずる。
- エラー型は `thiserror` で定義し、`anyhow` はバイナリ・CLI のトップレベルのみ許容する。
  ライブラリクレートで `anyhow` を使うことを禁ずる。
- エラーは呼び出し元に返すこと。握り潰すことを禁ずる。
- エラー処理の複雑さを理由に雑な互換レイヤーや暫定 API を導入することを禁ずる。
  複雑さは構造の誤りのシグナルであり、隠蔽対象ではない。
- **「この修正だけ通ればよい」という理由で本来のエラー境界を崩したり、責務をねじ曲げたり、例外的な分岐を埋め込んだりすることを禁ずる。**
  その種の対処はバグ修正ではなく、将来の崩壊点の先送りである。

## `unwrap()` 不要設計

`unwrap()` が必要に見える原因は、ほぼ常に **型が現実を正確に表現していないこと** にある。
`unwrap()` を書きたくなった瞬間に、型を直すことを考えよ。

### 原則：型が「値が存在する」ことを静的に証明する

`unwrap()` が必要になるのは、型に `Option` や `Result` が混入している場所である。
その `Option` / `Result` が本来不要であれば、型から取り除くのが正しい解答である。

#### ケース別対処法

**① 初期化前フィールドに `Option<T>` を使っている場合**

```rust
// ❌ 悪い例：初期化後は必ず Some であることを unwrap() で主張している
struct Server {
    listener: Option<TcpListener>,
}
impl Server {
    fn accept(&self) -> ... {
        self.listener.as_ref().unwrap().accept() // ← unwrap
    }
}

// ✅ 正しい解：TypeState で「未初期化」と「初期化済み」を型レベルで分離する
struct Server<S: ServerState> {
    inner: S,
}
struct Unbound;
struct Bound { listener: TcpListener }

trait ServerState: sealed::Sealed {}
impl ServerState for Unbound {}
impl ServerState for Bound {}

impl Server<Unbound> {
    fn bind(addr: SocketAddr) -> Result<Server<Bound>, BindError> { ... }
}
impl Server<Bound> {
    fn accept(&self) -> Result<TcpStream, AcceptError> {
        self.inner.listener.accept()
    }
}
```

**② コレクションの要素取得に `Option` を返す `get()` を使っている場合**

```rust
// ❌ 悪い例：インデックスが有効であることを知っているのに Option を受け取って unwrap する
let first = items.get(0).unwrap();

// ✅ 正しい解①：コレクション自体を「空でない」ことが保証された型で持つ
struct NonEmpty<T> {
    head: T,
    tail: Vec<T>,
}
impl<T> NonEmpty<T> {
    fn first(&self) -> &T { &self.head }
}
```

**③ 一度だけ初期化されるグローバル／シングルトンに `OnceLock<T>` 等を使っている場合**

```rust
// ❌ 悪い例：初期化済み保証をコメントと unwrap で表明する
static CONFIG: OnceLock<Config> = OnceLock::new();
fn get_config() -> &'static Config {
    CONFIG.get().unwrap()
}

// ✅ 正しい解：初期化と取得を分離せず、アプリケーション起動時に値を確定させて渡す
fn init_config(raw: RawConfig) -> &'static Config {
    CONFIG.get_or_init(|| Config::from(raw))
}
```

**④ 文字列パースの結果を「絶対成功するはず」として unwrap している場合**

```rust
// ❌ 悪い例
let addr: IpAddr = "127.0.0.1".parse().unwrap();

// ✅ 正しい解
use std::net::{IpAddr, Ipv4Addr};
const LOOPBACK: IpAddr = IpAddr::V4(Ipv4Addr::LOCALHOST);
```

**⑤ `HashMap::get` の結果を unwrap している場合**

```rust
// ❌ 悪い例
map.insert(key.clone(), value);
let v = map.get(&key).unwrap();

// ✅ 正しい解
let v = map.entry(key).or_insert(value);
```

**⑥ `Mutex::lock()` の結果を unwrap している場合**

```rust
// ❌ 悪い例
let guard = state.lock().unwrap();
```

この場合の本質的解答は `unwrap` の書き換えではなく、
**そもそも Mutex が必要な設計なのかを見直すこと** にある。

**⑦ 型変換（`TryFrom` / `TryInto`）の結果を unwrap している場合**

```rust
// ❌ 悪い例
let n: u8 = small_value.try_into().unwrap();

// ✅ 正しい解：変換失敗しない値域を最初から型で持つ
struct ByteValue(u8);
impl ByteValue {
    fn new(n: u32) -> Result<Self, OutOfRangeError> {
        n.try_into().map(Self).map_err(|_| OutOfRangeError)
    }
    fn get(&self) -> u8 { self.0 }
}
```

### `unwrap()` が絶対に許容されない文脈

以下の場所での `unwrap()` / `expect()` は理由の有無を問わず禁ずる。

- ライブラリクレートの公開 API（`pub` / `pub(crate)` の関数・メソッドの実装内）
- 非同期タスク（`tokio::spawn` のクロージャ内）
- エラー処理コード自体の内部
- ループのボディ内
- 削除移行中の暫定コード内

暫定コードだから雑に `unwrap` してよい、という発想を禁ずる。
中間状態が壊れていてよいのは **構造上の整合性を一時的に崩すこと** であって、
**設計思想そのものを壊してよい** という意味ではない。

### テストコードにおける `unwrap()`

テストコードであっても `unwrap()` の乱用は禁ずる。

```rust
fn test_parse() -> Result<(), Box<dyn std::error::Error>> {
    let result = parse("input")?;
    assert_eq!(result, expected);
    Ok(())
}
```

## 型設計

- **型でありえない状態を表現不可能にすること（Make Illegal States Unrepresentable）。**
  `bool` フラグや `Option<Option<T>>` で状態を表現することを禁ずる。
- **TypeState パターンを積極的に使用する。**
  オブジェクトのライフサイクル・状態遷移は型パラメータで表現し、
  無効な操作はコンパイルエラーとして排除すること。
- Newtype パターンを積極的に使用する。
  `String` や `u64` をそのまま渡し回すことを禁ずる。
- `Clone` のない設計を優先する。
  `#[derive(Clone)]` を反射的に付けることを禁ずる。必要性を説明できること。
- `Default` の自動 derive は、すべてのフィールドにおいてゼロ値が意味を持つ場合のみ許容する。
- フィールドに `Option<T>` を持つ場合は、その `Option` が何を意味するかを問い直すこと。
  「まだ初期化されていない」なら TypeState で解決する。
  「存在しないことがある」なら本当に Optional なのかドメインを再確認する。
- API 設計時は **今ある呼び出しやすさ** ではなく、
  **将来 100 箇所に増殖しても腐らない構造か** で判断すること。
- **現在の呼び出し側を最小変更で救済するために、型境界を曖昧化したり、本来分離すべき状態を同居させたり、暫定的な受け皿型を追加したりすることを禁ずる。**
  呼び出し側が壊れるなら、呼び出し側を直す。型を濁してはならない。

## 抽象

抽象は多ければよいのではない。
抽象は **現実の重複・現実の複数性・現実の境界** を表現するためにのみ存在すべきであり、
想像上の将来や書き味のために増やしてはならない。

### 実装が一つしかない段階での trait 導入を禁ずる

- **現実に複数実装が必要になっていない段階で、差し替え可能性だけを理由に trait を導入してはならない。**
- trait は実在する複数実装・明確な境界・テスト容易性ではなく構造純化に資する場合にのみ導入すること。
- 「将来 DB を差し替えるかもしれない」「将来 mock にしたいかもしれない」を理由に抽象化してはならない。
- 想像上の将来のための抽象は、互換レイヤーと同様に負債である。

### 意味を持つ分岐を `bool` 引数で表現してはならない

- `fn save(force: bool)` や `fn render(compact: bool)` のような API を禁ずる。
- `bool` 引数で意味分岐を埋め込むことは、ドメイン概念の隠蔽である。
- 分岐に意味があるなら、`enum`・Newtype・専用メソッド分割で表現すること。

```rust
// ❌ 悪い例
fn save(force: bool) -> Result<()> { ... }

// ✅ 正しい解
enum SaveMode {
    Normal,
    Force,
}
fn save(mode: SaveMode) -> Result<()> { ... }
```

### 汎用名・曖昧名の新設を禁ずる

- `Manager` / `Helper` / `Util` / `Common` / `Base` / `Tmp` / `Misc` のような、
  **責務を説明していない名前** を新設してはならない。
- 名称が責務を説明できない場合、その設計はまだ分解が足りていない。
- 曖昧名で分類できてしまう設計は、責務境界が曖昧である証拠として扱うこと。

## 所有権・借用

- `Arc<Mutex<T>>` は設計の敗北サインである。
  本当に共有可変状態が必要か再考すること。
  必要な場合は使用箇所にその根拠を明記すること。
- `clone()` の呼び出しはすべて意図的であること。
  コンパイルを通すための `clone()` は禁ずる。
- 生存期間（lifetime）を `'static` で誤魔化すことを禁ずる。
- 一時的な移行都合のために所有権設計を汚すことを禁ずる。
  借用が崩れるなら、局所修正ではなく構造から直すこと。

---

## グローバル状態

**グローバル可変状態を原則禁止する。**
値の受け渡しで表現できるものをグローバルへ逃がしてはならない。

- `static mut` は論外として禁ずる。
- `OnceLock` / `LazyLock` / グローバルキャッシュ / 暗黙シングルトン / process-wide mutable registry も原則禁止とする。
- 「どこからでも参照できる方が便利」は、依存方向と初期化責務の崩壊を意味する。
- グローバル状態が必要に見えた時点で、まず引数・所有者・初期化境界・アプリケーションルート設計を見直すこと。
- やむを得ずグローバルを使う場合でも、可変性・初期化順序・テスト独立性・破棄戦略を明示できなければ採用してはならない。

**グローバル状態は便利な共有ではなく、局所で表現できなかった責務の漏出である。**

---

## 非同期

- `async` を使う場合は **全体の非同期戦略** を明確にしてから導入すること。
  「とりあえず async にしておく」を禁ずる。
- `tokio::spawn` でタスクを生やしたら、そのタスクの終了を必ず管理すること。
  fire-and-forget を禁ずる（意図的な場合はコメントで明記）。
- `block_on` / `futures::executor::block_on` を async コンテキスト内で呼ぶことを禁ずる。
- 非同期タスク内での `unwrap()` / `expect()` は禁ずる。
- 移行中だからといって async 境界をごまかす互換ラッパーを増やすことを禁ずる。
  非同期戦略を変えるなら、境界からまとめて壊してまとめて作り直すこと。

---

## 境界・データ表現

外部入出力のための表現と、内部ドメインを表現する構造は本質的に異なる。
それらを同一視してはならない。

- **JSON / DB / API / CLI / 設定ファイル / 環境変数などの外部都合で作られた型を、そのまま内部ドメイン型として流通させることを禁ずる。**
- 外部境界用の DTO / request / response / record / raw config は、必ずドメイン型へ変換してから内部へ入れること。
- 外部都合による `Option`・nullable・欠損許容・文字列 enum・loosely typed fields を、内部の不変条件へ持ち込んではならない。
- DB スキーマや API レスポンスの都合で歪んだデータ形状を、そのままドメインモデルの責務として受け入れてはならない。
- シリアライズしやすさ、互換性、入出力都合は、内部モデルの純度より優先してはならない。

**外部境界は汚れていてよい。**
だが、その汚れを内部へ流し込んではならない。
汚れは境界で受け止め、変換し、ドメインへ入る時点で消えていなければならない。

## 設計方針：将来最適・削除ファースト

現在は **現状保存** ではなく **理想構造への収束** を優先する。
中途半端な現状維持は将来の負債になる。

- **新 API を追加したなら、同機能の旧 API を即座に削除する。**
- **正しくは「追加してから削除する」のではなく、削除してから必要最小限を再構築する」である。**
- **同一機能の別実装が同時にコードベース上に存在することを禁ずる。**
- **互換 shim・互換レイヤー・ラッパー・アダプターを残すことを禁ずる。例外はない。**
- `#[deprecated]` は原則使わない。
  旧設計の存在を許可する印として機能してしまうためである。
- 将来削除予定の型・トレイト・関数は新規コードで使用しない。
- 「今はまだ全部移行できないから両方残す」という判断を禁ずる。
- 「使っている箇所が多いから旧 API を残す」という判断を禁ずる。
  **使っている箇所が多いなら、なおさら先に消してコンパイラに全件洗い出させるべきである。**
- **ビルドを一時的に通すことそれ自体を目標にして、責務分割・型境界・モジュール境界を歪めることを禁ずる。**
  緑化は最終結果であって設計判断基準ではない。
- **最小変更で済ませるための継ぎ木・例外分岐・暫定保持・特殊ケース吸収が必要になった時点で、その修正方針は捨てること。**
  その場で一段上の抽象に戻り、必要なら全面的に作り直すこと。

### 移行中の公開 API 二重化禁止

**「移行中だから一時的に両方残す」という判断を禁ずる。**
これは最も頻繁に行われる違反であり、理由・文脈を問わず容認しない。

- 公開 API（`pub` / `pub(crate)` / crate 外から参照可能なシンボル）が二本存在する瞬間、
  呼び出し側は「どちらを使うべきか」を判断できなくなる。
- `v2_` / `_new` / `_legacy` などの接尾辞を持つシンボルが同時に存在する状態は、
  命名規則の崩壊であり設計の失敗である。
- **FFI ABI・ネットワークプロトコル・semver 境界など、いかなる外部境界を理由にした互換層も、
  実利用者が現実に存在しない限り認めない。**
  想像上の将来利用者のために現時点の設計を汚してはならない。

### test-only API の追加を禁ずる

**テストの都合でしか存在理由を説明できない API を追加してはならない。**
それはテスト容易性の問題ではなく、**本番設計がテスト不能な形に歪んでいる** ことのシグナルである。

- `#[cfg(test)]` の内外を問わず、**テスト専用の公開 API / 準公開 API（`pub` / `pub(crate)`）を追加することを禁ずる。**
- テストのためだけに内部状態の露出・可変化・初期化順序の緩和・依存差し替え用フックを後付けすることを禁ずる。
- `*_for_test` / `test_*` / `debug_*` / `expose_*` のような、
  **存在理由がテスト補助であることを示す API 名を新設することを禁ずる。**
- test-only API が必要になった時点で、**API を足すのではなく設計を見直すこと。**
  境界の切り方、責務分離、状態表現、依存注入の形が誤っている可能性を最優先で疑うこと。
- テストの書きやすさは、公開面の追加ではなく
  **純粋関数化・責務分離・副作用境界の明確化・型による状態保証** によって得ること。

**「テストを書くために API を増やす」のではなく、
「自然にテストできる構造へ設計を矯正する」のが正しい順序である。**

### 見た目だけの新 API 化を禁ずる

**実装を旧のままに据え置き、名前・シグネチャ・モジュールパスだけを新しく見せることを禁ずる。**

- 旧実装を内部で呼び出すだけの薄いラッパーを「新 API」と称して公開することを禁ずる。
- シグネチャを変えただけで内部ロジックが旧実装の複製・移植にとどまっている場合も同様に禁ずる。
- `#[inline] fn new_api(...) { old_impl(...) }` のような転送関数を経由して
  旧実装を温存することを禁ずる。
- 見た目だけの新 API 化は、削除ファースト原則が可視化するはずだった
  **本当の移行コストを埋没させる最悪の手法** である。

### 最小変更主義の明示的禁止

**レビューで最終採否を判断する以上、差分を小さく保つ必要は一切ない。**
**差分規模は設計判断の根拠にならず、むしろ理想構造を歪めずに到達できるかどうかだけを問う。**

**「差分が小さい」「今の呼び出し側をあまり触らずに済む」「とりあえずビルドだけ通る」ことを根拠に、汚い設計を選ぶことを禁ずる。**
これは単なるアンチパターンではなく、このコードベースにおける**最重要級の違反**である。

- 局所変更で収めるために責務境界を越境させることを禁ずる。
- 呼び出し側改修を避けるために、受け手側へ暫定分岐や過剰な入力許容を押し込むことを禁ずる。
- 一時的緑化のために、本来削除すべき概念や状態を残すことを禁ずる。
- 「後で直す前提」で不整合な抽象を追加することを禁ずる。
- 最小変更案と全面再設計案を比較して、前者が構造を汚すなら、**躊躇なく全面再設計を選ぶこと。**

**設計を汚してまで守る価値のあるビルドはない。**
**汚い継ぎ足しで通すくらいなら、壊してでも作り直した方が正しい。**

## 赤ビルド・壊れた中間状態の扱い

**ビルドが赤いままのコミットは許可する。**
**過程でコードベースが壊れることも許可する。**
ただしそれは「雑でよい」という意味ではなく、
**削除ファーストに忠実であるために、あえて中間整合性を犠牲にしている状態** でなければならない。

### 許可される赤状態

以下は明確に許可する。

- 旧 API を先に全削除した結果、呼び出し側が大量に壊れている状態
- 型の分割・TypeState 化・Newtype 化により、広範囲にコンパイルエラーが出ている状態
- 所有権設計の是正により、一時的に借用やライフタイムが未解決な状態
- モジュール再編・責務分割により import / パスが壊れている状態
- 新旧実装の並存を避けるため、旧実装削除後に新実装がまだ生え揃っていない状態
- **最小変更の応急処置を拒否した結果として、全面的な型・責務・境界の組み直しが必要になっている状態**

### 禁止される赤状態

以下は許可しない。

- 何を壊したのか自分で説明できない状態
- 削除ファーストと無関係な雑な破壊
- デバッグ残骸、コメントアウト放置、場当たり的な実験の混入
- 将来消すつもりの互換コードを追加したうえで赤くしている状態
- 設計を綺麗にするためではなく、単に面倒だから放置している赤状態
- **「今は赤いと困るから」という理由で、本来捨てるべき汚い構造を仮置きした状態**

### 赤コミットの原則

- **緑であることは品質保証ではない。**
  旧設計を抱えたまま通るビルドより、旧設計を消して大量に壊れているビルドの方が健全なことがある。
- **壊れ方は意図的でなければならない。**
  どの抽象を消したのか、何を露出させたのかを説明できること。
- **赤コミットは最終到達点ではなく、設計移行の途中経過である。**
  赤を許可する理由は、緑を守るために設計を妥協しないためである。
- **コミット粒度は設計上意味のある単位で切る。**
  「旧 API 削除」「型境界再定義」「呼び出し側全面移行」「新実装導入」「最終整合化」など、
  構造が読み取れる履歴にすること。

### テストを理由に旧 API を延命することを禁ずる

**API を更新したなら、旧 API はテスト都合を理由に一切残さない。**

- 「既存テストがまだ新 API に追随していない」は、旧 API 存置の理由にならない。
- 「テストヘルパーだけ旧 API に依存している」は、旧 API 存置の理由にならない。
- 「移行中の比較検証のため両方の API をテストから叩きたい」は、二重化禁止原則の例外にならない。
- 旧 API を残したままテストを通すことは、**設計を守った** のではなく
  **テストの都合で設計を汚した** に過ぎない。

正しい順序は常に以下である。

1. **旧 API を容赦なく削除する。**
2. 壊れたテストを通じて、旧 API 依存箇所を全露出させる。
3. テスト対象・テストヘルパー・フィクスチャを新設計へ全面移行する。
4. 最後にのみ、テストを再度緑化する。

**テストを守るために旧 API を残してはならない。**
**旧 API を消した結果としてテストが壊れるなら、その壊れ方こそが移行対象一覧である。**

### マージ・安定化

- 中間コミットや作業ブランチで赤いことは問題ない。
- 安定化ポイントでは最終的にビルド・テスト・lint を通すこと。
- ただし **安定化のために旧設計を戻すことは禁ずる。**
  緑に戻す手段は、削除を取り消すことではなく、新設計を完成させることだけである。
- **「安定化」とは、その時点で最小変更の汚い対処を混ぜて緑にすることではない。**
  安定化とは、理想構造が成立した結果として緑になることを指す。

## API 更新手順：削除を先にコミットしてよい

**新実装の設計が頭の中で固まったなら、次に動かす手は削除である。**
エディタを開いたら旧 API を消すことから始めよ。新しいコードは一行も書かなくてよい。

1. **旧 API・旧実装をすべて削除する。**
   新 API の実装がまだ存在しなくてよい。
   コードベースが壊れた状態になってよい。
2. `cargo check` / `deno check` を実行し、
   **コンパイルエラーの一覧を移行対象リストとして確定させる。**
3. 必要であれば、**この壊れた状態のままコミットしてよい。むしろ推奨される。**
   このコミットは「旧設計の除去が完了した」という強い履歴になる。
4. エラーが示す呼び出し箇所を順に新 API へ書き換えながら、新 API の実装を育てる。
5. 型・責務・境界が理想形へ揃うまで、途中で何度赤コミットしてもよい。
6. 最後にのみ、ビルド・テスト・lint を通して安定化する。

**この順序を逆にすること（新 API を先に書いて旧 API を後から削除する）を禁ずる。**
逆順では旧設計が見えなくなり、互換層や薄いラッパーが紛れ込む。
それは設計を進めたのではなく、**負債を別名で温存しただけ** である。

### 推奨コミット分解

削除ファーストを徹底する場合、以下のような分解を推奨する。

1. **delete:** 旧 API / 旧実装の全削除
2. **breakage surfacing:** 壊れた呼び出し側・型境界の露出
3. **type reshape:** TypeState / Newtype / モジュール境界の再定義
4. **rebuild:** 新 API / 新実装の導入
5. **stabilize:** テスト・lint・最終整合化

この分解の目的は、履歴から見ても
**何を消し、何を再定義し、何を新しく作ったのかが明瞭であること** にある。

### 例外的全面再構築の判断

**局所修正案が設計純度を損なうと判明した時点で、その修正は局所問題ではなく構造問題である。**
その場合は差分規模や既存コード量に関係なく、対象領域を**新しい設計で再構築し直す**こと。

- 「ここだけ一旦合わせる」は禁止。
- 「あとで整理する前提で今だけ通す」は禁止。
- 「変更範囲が広すぎるから今回は見送る」は禁止。

**最小変更で済まないことは失敗ではない。**
**最小変更で済ませようとして構造を汚すことこそが失敗である。**

## lint・警告

- lint を緩めることを禁ずる。
- `Cargo.toml` の `[lints.rust]` および `[lints.clippy]` に以下を必ず設定すること：

```toml
[workspace.lints.rust]
# ── 安全性 ───────────────────────────────────────
unsafe_code = "forbid"

# ── 設計純度 ─────────────────────────────────────
unreachable_pub = "deny"

# ── 無駄排除・保守性 ─────────────────────────────
unused = { level = "deny", priority = -1 }
dead_code = "deny"
missing_debug_implementations = "deny"

[workspace.lints.clippy]
# ── 全体方針 ─────────────────────────────────────
all = { level = "deny", priority = -1 }
pedantic = { level = "deny", priority = -1 }
nursery = { level = "warn", priority = -1 }

# ── 安全性: unsafe・panic・危険操作禁止 ───────────
undocumented_unsafe_blocks = "forbid"
unwrap_used = "forbid"
expect_used = "forbid"
panic = "forbid"
panic_in_result_fn = "forbid"
indexing_slicing = "forbid"

# ── 安全性: 変換の健全性 ─────────────────────────
as_conversions = "forbid"
cast_possible_truncation = "forbid"
cast_possible_wrap = "forbid"
cast_precision_loss = "forbid"
cast_sign_loss = "forbid"

# ── 設計純度: エラー設計・分岐の明確性 ────────────
result_large_err = "forbid"
error_impl_error = "forbid"
map_err_ignore = "forbid"
manual_ok_or = "forbid"
unreachable = "forbid"
wildcard_enum_match_arm = "forbid"

# ── 設計純度: 所有権・可変性の厳格化 ──────────────
mut_mut = "forbid"
needless_pass_by_value = "forbid"
unnecessary_mut_passed = "forbid"

# ── 無駄排除・保守性: 不要コピー・冗長記法排除 ────
redundant_clone = "forbid"
clone_on_ref_ptr = "forbid"
str_to_string = "forbid"
```

- `#[allow(...)]` を付ける場合は必ず理由をコメントで明記すること。

```rust
// ALLOW: proc-macro の展開によって生成されたコードのため dead_code 警告が誤検知する
#[allow(dead_code)]
```

- `#[allow(unused)]` `#[allow(dead_code)]` の理由なし使用は禁ずる。
- TypeScript コードは `deno lint` および `deno check` を CI で必ず実行すること。
- **ただし、移行途中の赤コミットの存在を理由に lint 方針自体を緩めることを禁ずる。**
  厳しい基準は最後まで保持し、安定化段階で再び満たすこと。

## テスト

- テスト専用 API を足してテストを書くことを禁ずる。
  テスト不能なら API を増やすのではなく、設計を分解し直すこと。
- 新機能にはテストを必ず書くこと。テストなしの安定化を禁ずる。
- `#[test]` は実装と同一ファイルの `mod tests` に書くこと。
- テストでも `unwrap()` の使用は禁ずる。
  `-> Result<(), Box<dyn std::error::Error>>` を返し `?` 演算子を使うこと。
- ハッピーパスのテストだけでなく、**エラーパス・境界値を必ずテストすること。**
- 設計移行の途中でテストが壊れること自体は許可する。
  ただし、それを理由に旧設計へ合わせたテスト互換コードを足すことを禁ずる。
- **テストを早く緑に戻すためだけの暫定的な設計妥協を禁ずる。**
  テストは理想構造の検証対象であって、理想構造を曲げる理由ではない。

## デバッグ・観測性

観測可能性は重要である。
しかしそれは、ドメイン中心部を汚してよい理由にはならない。

- **デバッグ補助のためだけに公開面・型・責務境界・初期化手順を変更してはならない。**
- `debug_*` / `trace_*` / `dump_*` / `inspect_*` のような、
  存在理由が観測補助である API を安易に公開してはならない。
- ログ出力のためにドメイン型へ一時的な可視化用フィールドや文字列変換都合を埋め込んではならない。
- 監視・メトリクス・トレースは副作用境界または外側のレイヤで付加すること。
- 観測の都合で内部構造を露出しなければならない場合、その設計はまず境界の切り方を疑うこと。
- **「今だけ調べたい」を理由に残骸コードを残してはならない。**
  一時観測コードは削除条件を持ち、不要になった時点で即削除すること。

**観測可能性は内部汚染ではなく、境界設計で確保する。**

## レビュー基準

レビューは単なるスタイル確認ではない。
レビューとは、理想構造からの逸脱を差し戻すための関門である。

以下のいずれかに該当する変更は、原則として reject する。

- **旧概念・旧責務・旧 API を別名で温存している**
- **実装が一つしかないのに trait や抽象層を追加している**
- **意味を持つ分岐を `bool` 引数・曖昧フラグ・ loosely typed input に押し込んでいる**
- **外部境界用の DTO / record / response / config 形状がそのまま内部へ流入している**
- **呼び出し側改修を避けるために、受け手側の型境界・責務境界・入力制約を濁している**
- **一時的緑化のために例外分岐・特殊ケース・互換コード・暫定保持を追加している**
- **削除より追加が先に来ており、旧設計の露出を隠している**
- **デバッグ・テスト・監視都合で公開面や責務境界を増やしている**
- **依存追加が設計純化ではなく、移行回避・実装短縮・局所都合のために行われている**
- **名称が責務を説明せず、曖昧な箱だけが増えている**

レビューで見るべきなのは、「動くか」だけではない。
**その変更が将来の理想構造へ近づけているか、あるいは負債を別名で温存しているか** である。

「今回は小さな変更だから」「後で整理するから」「一旦通したいから」は、
レビュー受理の理由にならない。
差分の小ささは品質の根拠ではなく、むしろ構造問題の隠蔽に使われやすいことを忘れてはならない。

## まとめ：禁止事項一覧

| 禁止事項 | 理由 |
|---|---|
| `unsafe` ブロック（根拠なし） | メモリ安全性の破壊 |
| `unwrap()` / `expect()` の原則使用 | 型設計の不備を隠蔽する |
| 公開 API 実装内の `unwrap()` / `expect()` | 呼び出し元が制御不能な panic |
| 非同期タスク内の `unwrap()` / `expect()` | サイレントドロップ・検知不能な失敗 |
| テストコードでの `unwrap()` | 診断不能な失敗メッセージ |
| 暫定コードだからという理由の雑な `unwrap()` | 設計思想の自己破壊 |
| 初期化フィールドに `Option<T>` を使い unwrap | TypeState で解決すべき |
| コレクション取得に `get().unwrap()` | NonEmpty 型等で解決すべき |
| 文字列リテラルの `.parse().unwrap()` | コンパイル時定数で表現すべき |
| ライブラリクレートでの `anyhow` | エラー型の情報損失 |
| フロー制御目的の `panic!` | 設計の放棄 |
| 反射的 `#[derive(Clone)]` | 所有権設計の形骸化 |
| 根拠なし `Arc<Mutex<T>>` | 共有状態設計の放棄 |
| コンパイル通しのための `clone()` | 借用設計の回避 |
| `'static` による lifetime 誤魔化し | 生存期間設計の破壊 |
| fire-and-forget `tokio::spawn` | タスクリークの温床 |
| グローバル可変状態・暗黙シングルトン | 初期化責務・依存方向の崩壊 |
| 同一機能の別実装の並存 | 二重管理・認知負荷・整合性破壊 |
| 移行中を理由にした公開 API の二重化 | 呼び出し側の混乱 |

## 最終宣言

このコードベースでは、**中間状態の綺麗さではなく、最終構造の綺麗さを守る。**
そのために必要なら途中で何度でも壊してよい。ビルドが赤いまま何度コミットしてもよい。

ただし許されるのは、
**将来の理想構造へ一直線に近づくための破壊だけ** である。

旧設計を残して緑を守るくらいなら、
旧設計を消して赤くなった方が正しい。

**追加より削除を先に。**
**互換より純化を先に。**
**一時安定より将来整合を先に。**
**最小変更で汚く延命するくらいなら、壊してでも、必要なら1からでも、理想構造として作り直す方が正しい。**
